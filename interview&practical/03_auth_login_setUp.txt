SECTION 1: WHY LOGIN + JWT? (CORE CONCEPT)
==========================================

Problem:

* How does server identify user after login?

Traditional Approach:

* Sessions (stored on server)

Modern Approach:

* JWT (stateless authentication)

Flow:
Login → Verify credentials → Issue token → Client stores token →
Client sends token in every request → Server verifies token

Why JWT?

* Stateless (no session storage)
* Scalable (works across distributed systems)
* Faster (no DB lookup per request)

Interview Insight:
“JWT enables stateless authentication, reducing server load and improving scalability.”

SECTION 3: LOGIN SERVICE (BUSINESS LOGIC)
=========================================

Function: loginUser({ email, password })

---

STEP 1: FIND USER

const user = await User.findOne({ email });

Why:

* Identify user in DB

If not found:

* Throw error → "Invalid credentials"

Security Note:

* Do NOT reveal whether email exists

---

STEP 2: PASSWORD VERIFICATION

bcrypt.compare(password, user.password)

Why:

* Compare hashed password securely

Concept:

* Passwords are stored hashed
* bcrypt compares plain vs hash

Security Insight:

* Prevents plain-text password exposure

---

STEP 3: GENERATE JWT TOKEN

jwt.sign(payload, secret, options)

Payload:
{
userId,
role
}

Secret:

* Stored in environment variable (JWT_SECRET)

Options:

* expiresIn: "7d"

Why:

* Token contains identity + authorization data

Important:

* Never store sensitive data in JWT payload

---

RETURN VALUE

{
user,
token
}

SECTION 4: JWT DEEP UNDERSTANDING (CRITICAL)
============================================

JWT Structure:
Header.Payload.Signature

1. Header:

* Algorithm (HS256)

2. Payload:

* userId
* role

3. Signature:

* Signed using secret key

---

KEY CONCEPTS:

1. Stateless:

* Server does NOT store session

2. Self-contained:

* Token carries user info

3. Signed:

* Cannot be modified without secret

---

SECURITY RISKS:

* Token theft → impersonation
* No automatic revocation
* Large payload → performance issues

---

BEST PRACTICES:

* Use short expiry
* Store secret securely
* Use HTTPS always
* Avoid sensitive data in payload

---

INTERVIEW INSIGHT:

“JWT is signed, not encrypted. Anyone can decode payload but cannot modify it without invalidating signature.”

SECTION 7: AUTH MIDDLEWARE (PROTECTED ROUTES)
=============================================

Purpose:

* Protect routes
* Allow only authenticated users

---

STEP 1: GET TOKEN

const authHeader = req.headers.authorization

Format:
"Bearer <token>"

---

STEP 2: VALIDATE HEADER

Check:

* Exists
* Starts with "Bearer "

---

STEP 3: EXTRACT TOKEN

const token = authHeader.split(" ")[1]

---

STEP 4: VERIFY TOKEN

jwt.verify(token, secret)

If valid:

* Returns decoded payload

---

STEP 5: ATTACH USER

req.user = decoded

Why:

* Used in downstream logic (RBAC)

---

STEP 6: HANDLE ERROR

Invalid token → 401 Unauthorized

---

INTERVIEW INSIGHT:

“Auth middleware decouples authentication from business logic and enables role-based authorization.”

=======================================

Example:

Usage:

if (req.user.role !== "admin") → deny access

SECTION 10: INTERVIEW QUESTIONS (FAANG LEVEL)
=============================================

Q1: Why JWT instead of sessions?
A:
Stateless, scalable, no server storage

Q2: Is JWT secure?
A:
Yes if used with HTTPS and proper secret management

Q3: Can JWT be revoked?
A:
Not directly → requires blacklist or short expiry

Q4: Difference between authentication and authorization?
A:
Auth = identity, Authorization = access control

Q5: What happens if token is modified?
A:
Signature verification fails

Q6: Where do you store JWT on frontend?
A:
HttpOnly cookies (preferred) or localStorage (less secure)

Q7: What is inside JWT payload?
A:
Non-sensitive user identifiers
